This class defines the templates for the communication and control backend. Every vehicle can have at least one backend at the same time. Every timestep, the methods ‚Äòupdate_state‚Äô and ‚Äòupdate_sensor‚Äô are called to update the data produced by the simulation, i.e. for every time step the backend will receive teh current state of the vehicle and its sensors. Additionally, the backend must provide a method named ‚Äòinput_reference‚Äô which will be used by the vehicle simulation to know the desired angular velocities to apply to the rotors of the vehicle. The method ‚Äòupdate‚Äô is called on every physics step and can be use to implement some logic or send data to another interface (such as PX4 through mavlink or ROS2). The methods ‚Äòstart‚Äô, ‚Äòstop‚Äô and ‚Äòreset‚Äô are callbacks that get called when the simulation is started, stoped and reset as the name implies.

initialize(vehicle)ÔÉÅ

A method that can be invoked when the simulation is starting to give access to the control backend to the entire vehicle object. Even though we provide update_sensor and update_state callbacks that are called at every physics step with the latest vehicle state and its sensor data, having access to the full vehicle object may prove usefull under some circumstances. This is nice to give users the possibility of overiding default vehicle behaviour via this control backend structure.

class pegasus.simulator.logic.backends.backend.BackendConfigÔÉÅ

Bases: ABC

This class represents the configuration for the communication and control backend. Every backend has a configuration, which is used to customize its behavior. The configuration is used to instantiate the backend at the simulation start.
